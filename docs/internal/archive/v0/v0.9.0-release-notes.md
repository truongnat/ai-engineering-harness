# v0.9.0 Release Notes

## Summary

`v0.9.0` is the **Stable Contract Freeze** release.

It documents and freezes the core capability pack contracts adopters and maintainers can rely on before the first stable `v1.0.0` release—without adding distribution automation, runtime adapters, or new product features.

## What Is Included

- stable contract index
- breaking change policy
- minimal install tier decision (deferred post-v1)
- frozen PACK.md contract
- frozen installed surface contract
- frozen target profile contract
- frozen goal artifact contract
- frozen validation behavior contract
- frozen runtime consumption contract
- frozen packaging and release contract
- frozen source pack vs target repo boundary contract
- v0.9.0 readiness and release scope documentation

## What Changed From v0.8.0

`v0.8.0` proved real target-repo dogfood (two repos, two runtimes, structural validation pass).

`v0.9.0` adds contract freeze artifacts on that evidence:

- eight frozen contract documents indexed in [stable-contract-index.md](stable-contract-index.md)
- explicit breaking-change rules for pre-v1 and post-v1
- decision to defer minimal install tier until after v1.0.0
- cross-links from authoring docs to frozen contracts
- tests reinforcing install surface, PACK headings, validation CLI modes, and target fixtures

No change to default `install.js` `exportPaths` or validator heading lists in this release—freeze documents current behavior.

## What Is Intentionally Not Included

- runtime adapters
- package publishing automation
- marketplace automation
- archive generation automation
- semantic validation
- minimal install tier implementation
- Scenario C as a release requirement
- server, database, or background systems

## Who Should Use This Release

Use `v0.9.0` if you:

- need a stable contract baseline before adopting or extending the pack for `v1.0.0`
- maintain forks or internal distributions and want frozen rules for breaking changes
- already dogfooded on `v0.8.0` and want documented contracts instead of implied behavior
- write tooling against `validate.js` CLI modes or install surface

## Known Limitations

- contracts are documented and tested structurally; manifest body and profile semantics are not validated in code
- packaging and archives remain manual
- dogfood evidence is from `v0.8.0` scenarios; no new Scenario C in this release
- `PACK.md` Included Surface body may list items not identical to `exportPaths`—maintainers align via checklist
