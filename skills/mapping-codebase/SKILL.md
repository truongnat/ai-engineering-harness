# mapping-codebase

## Purpose

Understand the repository structure, active artifacts, boundaries, and likely impact area before planning or execution.

## When To Use

- In unfamiliar codebases
- Before planning a change
- When impact scope is unclear

## When Not To Use

- When the affected area is already mapped and unchanged
- When the task is limited to updating a known artifact with no dependency implications

## Workflow

1. Read the current artifacts that define goal and state.
2. Inventory relevant directories, files, and entry points.
3. Identify dependencies, boundaries, and ownership lines.
4. Note likely impact zones and uncertainty.
5. Summarize the map in language that supports planning.

## Operating Principles

- Start broad, then narrow to the affected area.
- Prefer observed structure over guessed architecture.
- Separate facts, inferences, and unknowns.
- Focus on what planning or execution will need next.

## Output Format

- workspace map
- affected-area summary
- risk and unknown list

## Checklist Before Done

- [ ] Relevant artifacts were loaded
- [ ] The affected area is identified
- [ ] Key boundaries or dependencies are noted
- [ ] Unknowns are explicit instead of hidden
